# markdown-stream 稳定化评估

## 1. 当前结论

项目已经具备可用雏形，但还不能算“稳定可用的组件库”。

现状更接近：

- `core`：已经接近稳定内核
- `vue3`：功能最强，但仍处于功能迭代期
- `react`：可用 alpha 封装
- `vue2`：兼容性分支，未达到稳定维护标准

如果以“可以放心接业务、持续发版、改动不容易回归”为标准，当前距离稳定可用还差一轮完整的工程化收敛。

## 2. 已验证事实

我实际检查并验证了这些内容：

- 阅读了根 README、各子包 README、核心实现和关键测试
- 执行了根目录 `npm test`
- 执行了根目录 `npm run build`
- 单独执行了 `vue2` 的 `npm test` 和 `npm run build`

结果：

- 根构建通过
- `core` 测试通过
- `react` 测试通过
- `vue3` 测试失败
- `vue2` 可构建，但没有测试文件，`npm test` 直接失败退出

## 3. 分模块评估

### 3.1 core

成熟度：高

优点：

- 模块边界清晰，`parser / assembler / diff / session` 分层合理
- 流式状态模型明确：`start / streaming / done`
- `parse / write / flush / snapshot / reset / use` 接口已经比较完整
- 测试覆盖明显高于其他包
- 设计文档和实现基本一致

还需要补的点：

- 缺少性能基线和大输入场景验证
- 缺少更完整的边界行为说明，例如超长文本、异常 markdown、并发写入约束
- 目前使用“全量重解析 + diff”，适合 v1，但需要给出性能预期说明

结论：

`core` 是当前仓库里最接近稳定发布标准的部分，可以作为整个项目稳定化的锚点。

### 3.2 vue3

成熟度：中

优点：

- 功能最完整，文档最丰富
- 有 `MarkdownStream`、递归渲染组件、composable、自定义 token 扩展
- 还提供了 SFC 预览能力，说明探索比较深入

主要问题：

1. 测试当前是红的

失败集中在两类：

- 自定义组件注入 / 覆盖能力不稳定
- `VueSfcFenceRenderer` 的行为和测试不一致

这意味着 `vue3` 还没有进入“可稳定发版”的状态。

2. API 预期与测试预期不一致

从类型定义和 README 看，当前推荐方式是：

- 只用一个 `component`
- 在组件内部自己判断 `token.state`

但测试里仍然在验证：

- `done`
- `streaming`
- `start`

这种“按状态分别传组件”的旧接口能力。

这说明现在至少有一个问题：

- 要么实现退化了，但测试没更新
- 要么设计变了，但文档、测试、兼容策略没有一起收敛

这类问题比单纯 bug 更危险，因为会持续制造误解。

3. 包体积非常大

本次构建中，`vue3/dist/index.js` 约为：

- 3.1 MB
- gzip 后约 857 KB

对组件库来说，这个体积偏大。

根因基本来自 `VueSfcFenceRenderer` 将以下能力一起打进主包：

- `@vue/compiler-sfc`
- `vue.global.prod.js?raw`
- `vue3-sfc-loader`
- `@tailwindcss/browser?raw`

这类能力更适合作为可选扩展，而不是默认入口的一部分。

4. 测试失败暴露出注入契约不稳定

`MarkdownTokenNode.vue` 当前将注入值按 `ComputedRef` 读取，但测试里提供的是普通对象。

这反映出一个更深的问题：

- 组件内部依赖的 provide/inject 契约没有被清晰固定
- 测试作者和实现作者对这个契约的理解已经不一致

5. SFC 卡片模板和测试期望不一致

测试要求存在 `.ms-sfc-card-title`，但模板里并没有实际渲染 title 节点。

这说明：

- 要么实现漏掉了标题
- 要么测试基于旧版 UI

无论哪种，都说明 `vue3` 当前还处在重构后未完全收尾的状态。

结论：

`vue3` 是最值得优先稳定化的包，但在修复测试、统一 API 和拆分重量能力之前，不能称为稳定组件。

### 3.3 react

成熟度：中低

优点：

- 基础能力已经打通
- 构建通过
- 严格模式下异步流消费有专门测试
- API 设计和 `vue3` 保持了较高一致性

主要问题：

1. 测试覆盖明显不足

当前只有 1 个测试，主要验证：

- React StrictMode 下不会吃掉单次流

但还缺少：

- 自定义组件覆盖
- tokenTypes 注册
- 静态字符串渲染
- 流式切换到静态内容
- 错误处理
- cursor / className / data-state
- 嵌套 token 渲染

2. 更像“能用”，还不像“稳定”

现在它通过测试，不代表成熟度高，而是代表测试面窄。

3. SSR / framework 集成风险未验证

当前实现主要依赖 `useEffect` 驱动解析和流消费，这对 CSR 没问题，但如果要走：

- Next.js
- RSC 场景
- SSR hydration

就还需要专项验证。

结论：

`react` 现在可以作为试用包，但不建议对外宣称为成熟稳定版本，除非先补齐测试和使用边界说明。

### 3.4 vue2

成熟度：低

优点：

- 基础构建还能过
- API 表面上与 `vue3` 对齐

主要问题：

1. 没纳入根 workspace

根 `package.json` 的 workspaces 只包含：

- `core`
- `vue3`
- `react`
- `playground`

不包含 `vue2`。

这意味着：

- 根级 `build/test` 不会覆盖它
- 它实际上不在统一维护闭环内

2. 没有测试

单独执行 `vue2` 的 `npm test`，结果是：

- `No test files found`

对组件库来说，这基本意味着它还不具备稳定发布资格。

3. Vue 2 本身已经 EOL

即使它能工作，也不适合再当作主支持方向。

结论：

`vue2` 应该二选一：

- 要么明确降级为“实验/兼容包”
- 要么补进正式维护链路

现在这种状态最危险，看起来像正式包，实际上没有稳定性保障。

## 4. 距离“稳定使用的组件”还差哪些能力

这里按“必须补齐”和“建议补齐”两层来分。

### 4.1 必须补齐

#### A. 恢复测试可信度

目标：

- 所有正式支持的包测试全绿
- 测试内容反映真实公开能力

需要完成：

- 修复 `vue3` 当前失败测试
- 删除或重写已经不符合现状的测试
- 为 `react` 补齐核心场景测试
- 明确 `vue2` 是否继续维护；继续维护就必须补测试

如果这一步不做，后续任何重构都无法证明没有回归。

#### B. 统一公开 API 契约

要明确回答这些问题：

- `components` 是否支持响应式更新？
- 是否支持 `done / streaming / start` 这种按状态组件写法？
- 旧格式 `Record<string, Component>` 会维护多久？
- `content / source / stream` 三套入口是否长期并存？
- `flush()` 是否算正式公开 API？

目标不是让 API 更多，而是让 API 更稳定、更少歧义。

#### C. 拆分重量级能力

建议把 `VueSfcFenceRenderer` 这类能力从主入口拆出去，例如：

- 单独入口
- 可选插件
- playground/demo only

否则会持续带来：

- 包体积过大
- 浏览器兼容面变复杂
- 测试成本提升
- 文档心智负担上升

#### D. 建立正式工程护栏

至少需要补齐：

- CI
- lint
- coverage
- 统一构建和测试矩阵
- 发版流程

没有这层护栏，组件“现在能用”不等于“后续还能稳定可用”。

#### E. 文档收敛

当前文档问题包括：

- 根 README 版本号过期
- 根 README 包列表不完整
- 文档重点偏 Vue 3，React/Vue 2 信息不对称
- 一些测试预期和现文档表达已经分叉

稳定组件库需要做到：

- README 能准确说明支持矩阵
- 每个子包文档都对应当前真实能力
- 示例代码可直接运行

### 4.2 建议补齐

#### A. 明确性能与规模边界

目前 `core` 采用全量重解析 + diff，v1 合理，但建议补上：

- 大文档性能测试
- 高频 chunk 写入测试
- chunk 粒度建议
- 内存占用观察

#### B. 明确 SSR/CSR 支持边界

特别是 React / Vue 3，要明确：

- 是否支持 SSR
- 是否支持 hydration 后继续流式更新
- 哪些能力仅浏览器可用

#### C. 增加兼容性说明

建议至少明确：

- Node 版本
- Vue / React 最低版本
- 浏览器兼容范围
- 是否依赖 `AsyncIterable`

#### D. 发布规范化

建议补：

- changelog
- semver 策略
- 破坏性变更说明
- alpha/beta/stable 标记规则

## 5. 推荐的稳定化优先级

这里按 P0 / P1 / P2 拆。

### P0：必须先做

1. 修复 `vue3` 测试失败
2. 对齐 `vue3` 的实现、测试、类型和 README
3. 明确 `vue2` 命运：纳入正式维护，或明确降级
4. 给 `react` 补一轮基础测试
5. 建立根级 CI，至少覆盖 build + test

这是“从原型走向稳定”的最低门槛。

### P1：紧接着做

1. 拆分 `VueSfcFenceRenderer` 出主包
2. 梳理并精简公开 API
3. 增加 lint / coverage / release 流程
4. 更新根 README 和各子包 README
5. 明确支持矩阵和非目标范围

这是“变成可维护组件库”的关键阶段。

### P2：进一步增强

1. 增加性能基准测试
2. 增加 SSR / framework 集成验证
3. 提供更清晰的扩展模式示例
4. 加强 playground，作为回归验证入口

这是“从稳定可用走向成熟易推广”的阶段。

## 6. 我建议的落地顺序

如果让我直接排执行顺序，我建议这样做：

1. 先处理 `vue3`

- 修复注入契约问题
- 修复 `VueSfcFenceRenderer` 的模板/测试不一致
- 清理失效测试或补回兼容实现

2. 再补 `react`

- 不急着加新功能
- 先把核心测试补够

3. 决策 `vue2`

- 如果继续维护，就纳入 workspace 和 CI
- 如果不继续维护，就文档上明确标成兼容/实验

4. 最后做工程化

- CI
- lint
- coverage
- release
- changelog

5. 再做包体积优化和高级能力拆分

- 尤其是 SFC 预览能力

## 7. 适合对外的当前表述

如果现在要对外描述项目状态，我建议用更保守的说法：

> `@markdown-stream/core` 已基本可用；`@markdown-stream/vue3` 处于快速迭代期；`@markdown-stream/react` 为早期可用版本；`@markdown-stream/vue2` 为兼容性尝试，暂不建议作为稳定依赖。

这个表述比“组件库已经稳定可用”更贴近实际，也更能保护后续信誉。

## 8. 最终判断

### 已经具备的基础

- 有明确的核心模型
- 有跨框架封装方向
- 有一定测试基础
- 有 playground 和示例
- 有较强的 LLM 流式场景针对性

### 还没达到稳定组件库的原因

- `vue3` 主力包测试仍未收敛
- 多包维护状态不一致
- API 契约仍在漂移
- 缺少正式工程护栏
- 主包中混入了过重的扩展能力

### 结论

这个项目不是“还没做出来”，而是“已经做出来了第一版，但还没完成稳定化”。

距离稳定可用，不差重写，差的是：

- 收敛
- 校准
- 护栏
- 分层

## 9. 下一步可直接执行的任务清单

如果要立刻开工，我建议先开这 8 个任务：

1. 修复 `vue3` 的 `MarkdownTokenNode` 注入契约问题
2. 修复 `vue3` 的 `VueSfcFenceRenderer` 标题与测试不一致问题
3. 决定是否保留 `done / streaming / start` 专属组件 API
4. 更新 `vue3` 的测试，使其只验证正式支持能力
5. 为 `react` 增加至少 6 到 10 个关键场景测试
6. 决定 `vue2` 是正式支持还是降级为实验包
7. 增加根级 CI
8. 更新根 README 的包列表、版本说明和状态说明
